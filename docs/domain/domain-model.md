## 2. 核心领域对象

### 2.1 用户 User

用户是平台内执行操作的主体。

已确认用户相关规则：

- 用户可以属于多个组。
- 用户可以在所属或被授权组范围内访问多个组的模板/母版。
- 访问账号可以是 service account，也可以是 user account。
- API 调用时需要读取访问账号的 AD Group。
- 后台治理与管理界面本迭代采用本地管理账户认证。
- 本地管理账户至少包含用户名、用户姓名、用户邮箱、密码哈希、角色集合、授权分组范围摘要和启停状态。
- 测试管理账户用户名采用八位工号格式。

### 2.1.1 管理认证会话 Management Authentication Session

管理认证会话用于承载后台治理与管理界面的登录后身份上下文。

已确认属性：

- 会话标识。
- 认证来源（当前迭代为 `LOCAL`）。
- 关联用户摘要。
- 角色集合。
- 授权分组范围摘要。
- 签发时间。
- 过期时间或失效状态摘要。

已确认规则：

- 登录成功后，后台治理与管理界面基于已认证管理会话加载角色与分组范围。
- 会话失效后，后台治理与管理界面需要回到登录入口，不继续沿用失效前的前端角色上下文。
- 当前迭代需要为未来公司 SSO 集成保留扩展边界，但不把 SSO/OIDC 集成声明为已交付能力。

### 2.1.2 管理用户 ManagementUser（生命周期与不变式）

ManagementUser 是后台治理与管理界面的本地账户对象，也是平台授权权威源的承载者。本地账户库长期作为授权权威源，未来 SSO 仅负责认证，不负责角色与组范围映射（见 [ADR 0036](../adr/authorization-security/0036-local-account-store-authorization-authority.md)）。

已确认属性：

- 用户标识。
- 用户名（八位工号格式，全局唯一）。
- 显示名。
- 邮箱。
- 密码哈希（Argon2id，仅存哈希，不存明文）。
- 认证来源（当前迭代为 `LOCAL`）。
- 角色集合。
- 被授权组范围（`authorizedGroupCodes`）。
- 启停状态（`enabled`）。
- 创建时间、更新时间（UTC）。
- 逻辑删除标记（`deletedAt`）。

已确认生命周期与规则：

- 用户全生命周期覆盖创建、编辑（显示名、邮箱）、分配角色、分配被授权组范围、停用/启用、逻辑删除、重置密码。
- 启停状态采用 `ENABLED` 与 `DISABLED` 两个状态；逻辑删除采用 `deletedAt` 标记。
- 重置密码采用管理员传入新口令模式：平台只存 Argon2id 哈希，不返回一次性临时口令，不在响应、日志或审计中回显口令明文或哈希。
- 已停用用户不可登录；已逻辑删除用户不可登录、不再参与授权判定，且不可被重新启用（需新建用户）。
- 用户的角色集合与被授权组范围始终由本地用户/分组管理维护，不被外部身份提供方创建、修改或删除。
- 用户被授权组范围内的每个组编码必须引用一个存在且未停用的 BusinessGroup。

不变式：

- 用户名全局唯一；创建重复用户名返回 `409 USERNAME_ALREADY_EXISTS`。
- 分组管理员创建/编辑用户时，目标用户被授权组范围必须 ⊆ 分组管理员自身被授权组范围（否则 `403 GROUP_SCOPE_OUT_OF_RANGE`）。
- 分组管理员只能分配运营类角色（`MASTER_DESIGNER`、`TEMPLATE_AUTHOR`、`TEMPLATE_TESTER`、`TEMPLATE_APPROVER`），不得分配 `GLOBAL_ADMIN`、`AUDIT_ADMIN`、`GROUP_ADMIN`（否则 `403 ROLE_ASSIGNMENT_NOT_ALLOWED`）。
- 逻辑删除用户仅全局管理员可执行（否则 `403 USER_DELETE_NOT_ALLOWED`）。

### 2.2 角色 Role

已确认系统角色包括：

- 管理员。
- 母版设计人员。
- 模板编排人员。
- 测试人员。
- 审批人员。
- 审计管理员。

管理员范围包括：

- 全局管理员。
- 分组管理员。

全局管理员拥有系统最大权限。

分组管理员的管理范围由显式授权的一个或多个组决定，不要求等同于用户自身所属组；分组管理员仅在被授权组范围内拥有管理员权限。

API 管理由全局管理员和分组管理员承担，不设置独立 API 管理员角色。

测试人员和审批人员按分组配置。审计管理员用于查看审计记录。

非主责角色采用最小权限；除已确认职责外，不获得额外母版/模板编辑权限。测试人员只执行测试通过/不通过判定，审批人员只执行审批通过/不通过判定，API 调用方只调用已授权 API。

登录起点角色旅程重构（T01 首波）已确认规则：

- 首波角色范围为 `GLOBAL_ADMIN`、`GROUP_ADMIN`、`TEMPLATE_AUTHOR`。
- 登录鉴权边界本迭代采用真实本地管理账户认证，并为未来 SSO/OIDC 集成保留扩展边界。
- 新旧导航并行切换由特性开关控制，不引入新的持久权限对象。
- 首波角色旅程链路基线：登录 -> 角色落地页 -> 角色首个关键任务。

待确认（不作为已确认领域规则）：

- `TEMPLATE_AUTHOR` 与现有“母版设计人员”在身份映射层面的最终对齐策略。
- 角色关键任务完成时间指标的量化口径和阈值。

### 2.2.1 路由可见性与禁止访问判定 Route Visibility and Forbidden Decision

路由可见性与禁止访问判定用于承载登录后导航可见性和路由访问控制的领域语义。

已确认属性：

- 逻辑路由标识（`routeKey`）。
- 角色可见性集合。
- 分组范围约束摘要。
- 判定结果（`ALLOWED` 或 `FORBIDDEN`）。
- 判定追踪标识（`traceId` 或 `auditId`）。

已确认规则：

- 路由可见性按角色和分组范围联合判定，任何入口都不得绕过对象范围检查。
- 已登录但无目标路由权限时，判定结果为 `FORBIDDEN`，并返回统一无权访问反馈。
- `FORBIDDEN` 判定不得泄露未授权资源存在性细节、未授权组详情或敏感配置明文。
- `FORBIDDEN` 判定需要记录安全审计摘要，至少包含主体摘要、入口路由标识、判定结果和拒绝原因码。
- 新旧导航并行期间，特性开关关闭时沿用旧导航判定规则，不将新导航可见性外溢到旧导航。

首波逻辑路由可见性基线：

| `routeKey` | GLOBAL_ADMIN | GROUP_ADMIN | TEMPLATE_AUTHOR |
| --- | --- | --- | --- |
| `route.global-governance-home` | 允许 | 禁止 | 禁止 |
| `route.group-governance-home` | 允许 | 允许 | 禁止 |
| `route.template-authoring-home` | 允许 | 允许 | 允许 |
| `route.api-policy-management` | 允许 | 允许（被授权组范围内） | 禁止 |
| `route.audit-console` | 允许 | 允许（被授权组范围内） | 禁止 |

### 2.3 分组 BusinessGroup

分组用于人员、模板、母版访问范围的隔离，是一等可运营对象（不再仅为种子数据）。

已确认属性：

- 分组标识。
- 分组编码（`groupCode`，全局唯一，创建后不可修改）。
- 显示名。
- 维度（`dimension`，取值 `BUSINESS_LINE` 业务条线 / `DEPARTMENT` 部门，创建时确定）。
- 启停状态（`enabled`）。
- 创建时间。

已确认规则：

- 母版设计人员需要分组。
- 模板编排人员需要分组。
- 母版本身需要严格按组隔离。
- 模板本身需要分组隔离。
- 模板只能在所属或被授权组范围内使用和维护。
- 不同组之间不允许复用模板。
- 分组隔离采用混合隔离模型。
- 当前确认的分组维度包括业务条线、部门/团队，并固化为 `dimension` 取值 `BUSINESS_LINE` 与 `DEPARTMENT`。
- 用户可以属于多个组。
- 分组管理员可管理被显式授权的一个或多个组。
- 母版只能在所属或被授权组范围内查看、使用和维护。
- 不同组之间不允许复用母版。

已确认生命周期与规则（一等对象）：

- 分组支持创建、编辑（仅显示名）、停用、启用；分组不提供逻辑删除或物理删除。
- 分组创建、编辑（显示名）、停用、启用仅全局管理员可执行；分组管理员对分组只读，仅可见被授权组范围内分组。
- 编辑操作只修改显示名，不修改 `groupCode` 或 `dimension`。
- 已停用分组不可作为新的用户被授权组范围分配目标，也不可作为新的母版/模板归属组使用；停用不影响已锁定的发布版本与历史审计。

不变式：

- `groupCode` 全局唯一；创建重复编码返回 `409 GROUP_CODE_ALREADY_EXISTS`。
- 分组不存在或不在请求方可见范围内时返回 `404 GROUP_NOT_FOUND`，不泄露存在性。
- 分组管理员对分组的任何写操作均 fail-closed，返回 `403 GROUP_MANAGEMENT_NOT_ALLOWED`。

### 2.4 AD Group

AD Group 用于 API 调用授权。

已确认规则：

- 一个模板可以绑定多个 AD Group。
- 不同模板可以绑定不同 AD Group。
- API 调用时需要读取访问账号的 AD Group。
- 访问账号可以是 service account，也可以是 user account。
- AD Group 解析规则适用于所有需要 AD Group 授权的 API 操作，包括生成、批量生成、异步任务查询、异步任务取消、下载取文件、API 契约查看和可调用版本列表。
- AD Group 成功解析结果按 `accessAccount` + `environment` 缓存 5 分钟；不缓存解析失败结果。
- AD Group 解析失败时，如果存在未过期缓存，则使用未过期缓存继续授权；如果不存在未过期缓存，则返回 `503 AD_GROUP_RESOLUTION_FAILED`，`retryable=true`。
- AD Group 授权不得使用过期缓存兜底；过期缓存不能作为授权依据。
- API 管理中的 AD Group 授权配置变更立即生效，并清理相关授权缓存；不等待 5 分钟缓存自然过期。
- 目录中的 AD Group 成员变更在目录同步完成且平台缓存过期后生效；平台需要在 API 契约或管理界面说明最多可能存在 5 分钟平台缓存延迟，不承诺消除外部目录同步延迟。
- AD Group 解析、缓存命中、缓存失效、解析失败和授权拒绝需要记录审计摘要；审计不得记录完整 AD Group 成员清单或未授权组详情。

### 2.5 母版 Master Document

母版是 DOCX 格式的基础文档框架。

**管理 UI 导航（2026-06-24）：** 侧栏为 **Masters** / **Templates** 包列表；包详情内展示修订线/版本线。见 `docs/product/catalog-navigation-ux.md`。母版在 v1 数据模型中为单资产 + 轻量审核状态，非模板式多发布版本实体。

已确认属性：

- DOCX 文件。
- 锚点集合。
- 基础版式。
- 样式目录。
- 所属分组。
- 审核状态。
- 审核记录。
- 变更说明。
- 影响分析摘要。

已确认规则：

- 母版以锚点形式存在。
- 母版用于确定文档的基础框架。
- DOCX 母版里的锚点由母版设计人员在 DOCX 中预先定义。
- 母版不需要类似模板的完整生命周期。
- 母版不需要可用/停用状态，只作为 DOCX 资产存在。
- 母版需要轻量审核状态，状态集合为草稿、待审核、审核通过、审核不通过；该状态集合不构成类似模板的完整生命周期，也不引入可用/停用或发布版本。
- 母版只能在所属或被授权组范围内查看、使用和维护。
- 不同组之间不允许复用母版。
- 母版更新后，不自动影响已有模板。
- 母版设计人员负责上传/创建 DOCX 母版、维护锚点、修改基础版式、提交母版审核。
- 母版设计人员可创建母版、删除母版、更新母版基础信息；此前在母版语境下提到的“创建模板、删除模板、更新模板基础信息”均指母版。
- 分组管理员可管理被授权组范围内的全部母版常规操作，包括上传/创建 DOCX 母版、维护锚点、修改基础版式、提交母版审核、删除母版、更新母版基础信息，并查看母版审核、变更和影响分析审计。
- 母版审核由管理员执行；全局管理员可审核全部母版，分组管理员可审核被授权组范围内母版。
- 提交母版审核前必须执行锚点完整性校验并填写变更说明；锚点完整性校验失败时不得提交审核，变更说明进入审核摘要和审计记录。
- 母版审核通过后才允许被新建或更新模板引用；草稿、待审核和审核不通过的母版不得被模板引用。
- 母版审核不通过后回到草稿，并保留审核记录；修改后可重新提交审核并生成新的审核记录。
- 母版变更需要影响分析，影响分析至少包含引用模板清单和重新测试提示。
- 母版审核通过后形成可供模板引用的 DOCX 母版样式目录。
- 母版样式目录需要标明可用样式名称、适用节点类型和渲染用途。
- 母版样式目录变更后，不自动影响已发布模板；模板升级母版样式目录或调整样式映射时，必须执行影响分析，并重新经过测试、审批和发布流程。

### 2.6 锚点 Anchor

锚点是母版中可被模板编排内容填充的位置。

锚点内容类型需要支持：

- 静态文本。
- 变量占位符。
- 富文本段落。
- 表格。
- 图片。
- 条款片段。
- 签章/盖章占位。
- 二维码/条码。
- 附件清单。

已确认规则：

- 锚点由母版设计人员在 DOCX 中预先定义。
- 模板编排负责锚点内容编排。
- 生成文档时如果母版锚点缺失，则生成失败并提示错误。

### 2.7 模板 Template

模板是基于母版进行锚点内容编排后形成的文档生成资产。

已确认属性：

- 所属母版。
- 所属分组。
- 模板状态。
- 开发版本集合。
- 发布版本集合。
- 模板变量。
- 编排规则。
- 测试数据集。
- 批量测试摘要。
- 样例覆盖率和模板综合覆盖率摘要。
- 测试生成记录。
- 生成预览。
- 最终 DOCX/PDF 产物引用。
- 预览对比摘要。
- 变更差异摘要。
- 发布前检查清单。
- 审批摘要。
- 关联的 API 管理配置。

已确认规则：

- 模板本身需要分组隔离。
- 模板只能在所属或被授权组范围内使用和维护。
- 不同组之间不允许复用模板。
- 普通模板删除仅由全局管理员执行；分组管理员、模板编排人员及其他角色不能删除普通模板。
- 模板废弃由全局管理员和分组管理员执行；全局管理员可废弃全部模板，分组管理员可废弃被授权组范围内模板。
- 普通模板基础信息更新由全局管理员和分组管理员执行；全局管理员可更新全部普通模板基础信息，分组管理员可更新被授权组范围内普通模板基础信息。
- 分组管理员可在被授权组范围内执行模板常规操作，包括选择母版创建模板、配置锚点内容、配置模板变量、配置条件/循环规则、测试生成 DOCX/PDF、提交测试/审批、发布/停用模板。
- 母版设计人员可以完整参与模板创建与编排，权限与模板编排人员一致，包括选择母版创建模板、配置锚点内容、配置模板变量、配置条件/循环规则、测试生成 DOCX/PDF、提交测试/审批、发布/停用模板。
- 模板编排人员负责选择母版创建模板。
- 模板编排人员负责配置锚点内容、模板变量、条件/循环规则。
- 模板编排人员负责测试生成 DOCX/PDF、提交测试/审批、发布/停用模板。
- 模板编排与 API 管理需要严格区分。AD Group 授权配置和 DOCX/PDF 动态加密配置不属于模板编排或模板提交功能，只在 API 管理中配置。

### 2.8 模板变量 Template Variable

模板变量用于动态填充文档内容或参与规则计算。

变量类型需要支持：

- 文本。
- 数字。
- 金额。
- 日期。
- 枚举。
- 布尔。
- 列表/数组。
- 对象/嵌套结构。

变量/规则还需要支持：

- 计算表达式。
- 聚合函数。
- 复杂逻辑计算。
- 复杂逻辑计算只能基于 API 入参和模板内配置计算，不允许调用外部数据/API。

### 2.9 编排规则 Composition Rule

编排规则控制锚点内容如何渲染。

需要支持：

- 必填/非空校验。
- 条件显示。
- 循环渲染。
- 默认值。
- 格式化。
- 简单计算。
- 枚举映射。
- 跨字段校验。

### 2.9.1 结构化文档片段 Structured Document Fragment

结构化文档片段是模板正文、富文本段落、条款引用和可复用业务文本的受控内容单元。

已确认属性：

- 块级结构。
- 行内文本片段。
- 变量节点。
- 编号层级。
- 条件块。
- 循环块。
- 样式引用。
- 条款或内容模块引用。
- 表格组件引用。
- 图片引用。
- 二维码/条码引用。
- 签章/盖章引用。
- 附件清单引用。
- 有限直接格式属性。

已确认规则：

- 模板正文内容主要通过模板结构、结构化文档片段、条款或内容模块、表格组件、变量和规则组合生成。
- 富文本源数据采用受控结构化模型，不以任意 HTML/CSS 作为发布契约。
- HTML 可以作为编辑或展示形式，但不得替代发布版本锁定的结构化内容契约。
- 结构化内容需要映射到支持的 DOCX 构造，包括样式、段落、文本片段、编号、表格、图片、签章/盖章、附件和生成文档元数据。
- 不支持的富文本构造需要在发布前检查中识别，不得静默降级。
- v1 支持的块级节点包括章节/标题、段落、列表、条件块、循环块和表格组件引用。
- v1 支持的行内节点包括文本片段、变量节点、强调、下划线和换行。
- v1 支持的引用节点包括条款或内容模块引用、图片引用、二维码/条码引用、签章/盖章引用、附件清单引用和样式引用。
- v1 样式引用来自审核通过的 DOCX 母版样式目录；模板只能引用当前可访问、已审核通过母版提供的样式。
- 不适用于当前节点类型的样式不得被发布候选引用。
- 模板编排人员可以选择命名样式，也可以在平台白名单内配置有限直接格式。
- v1 有限直接格式仅包括字体、字号、文本颜色、行距、段前段后间距、首行缩进、左缩进和右缩进。
- 有限直接格式必须作为受控结构化属性保存，不得以任意 CSS、任意 Word 脏样式或脚本形式进入发布契约。
- 有限直接格式不得修改页眉页脚、页边距、纸张、全局版式、复杂分栏、浮动或绝对定位。
- v1 明确不支持任意 HTML/CSS、脚本、绝对定位、浮动、复杂分栏、在线任意修改页眉页脚/页边距/全局版式，以及未被平台模型识别的 Word 脏样式进入发布版本。
- Word 或 HTML 粘贴内容必须清洗并转换为受控结构化节点后才能进入模板；不可识别内容不得作为原始格式直接进入发布契约，应按风险进入阻断项或警告项。
- Word 或 HTML 粘贴内容只允许转换为 v1 已确认的受控结构化节点和受控结构化属性。
- 粘贴自动转换范围包括段落、章节/标题、列表、文本片段、强调、下划线、换行、样式引用和白名单内有限直接格式。
- 粘贴内容中的简单表格可转换为表格组件草稿；复杂表格或无法可靠识别结构的表格不得直接进入模板。
- 粘贴内容中的本地图片可转换为图片引用草稿；外链图片不得直接进入模板。
- 粘贴内容不得自动识别为签章/盖章、二维码/条码或附件清单；这些内容必须通过对应受控节点或组件插入。
- 脚本、宏、嵌入对象、iframe、外部资源、绝对定位、浮动、复杂分栏、未识别 Word 样式、无法映射的变量、无法映射的样式和无法可靠识别的表格结构，均阻断本次粘贴进入模板。
- 不可识别内容不得以原始 Word、HTML、CSS 或二进制片段形式持久化到发布契约。
- 粘贴清洗完成后，必须向用户展示清洗摘要，至少区分已转换内容、已删除内容、警告项和阻断项。
- 用户必须可以取消本次粘贴，或在粘贴后撤销到粘贴前状态。
- 粘贴清洗摘要和阻断项需要进入模板编辑或发布前检查的可追溯信息；摘要不得保存被清洗内容中的敏感明文或原始源内容。

### 2.9.2 条款或内容模块 Clause or Content Module

条款或内容模块是可复用、可版本化、可审批的业务内容资产，用于承载标准条款、可选条款、标准段落或其他受控业务文本。

已确认属性：

- 内容结构。
- 版本历史。
- 版本状态。
- 审批状态或审批记录。
- 适用范围。
- 所属分组或可访问范围。

已确认规则：

- 条款片段升级为可复用、可版本化、可审批的条款或内容模块。
- 条款或内容模块作为一等领域对象继续建模。
- 模板和结构化文档片段可以引用条款或内容模块。
- 条款或内容模块 v1 采用独立版本审批生命周期。
- 条款或内容模块版本状态包括草稿、待审批、已批准、已停用和已废弃。
- 条款或内容模块审批不通过后回到草稿。
- 每个条款或内容模块版本必须独立审批；未批准、已停用或已废弃的版本不得被新的模板发布候选引用。
- 条款或内容模块不新增专门维护角色，复用现有全局管理员、分组管理员、母版设计人员、模板编排人员、测试人员和审批人员角色。
- 母版设计人员和模板编排人员可在所属或被授权组范围内创建条款或内容模块、编辑草稿、创建新版本并提交审批。
- 审批人员按分组执行条款或内容模块审批通过或审批不通过判定；分组管理员只在异常处理或补救场景中作为例外管理员干预审批判定。
- 条款或内容模块默认按所属组隔离；全局管理员和分组管理员可配置授权组共享范围。
- 模板只能引用同组或已授权共享范围内的条款或内容模块。
- 模板发布版本必须锁定引用的具体已批准条款或内容模块版本。
- 条款或内容模块新版本发布后，不自动影响已发布模板；模板升级引用新模块版本时，必须执行影响分析，并重新经过测试、审批和发布流程。
- 条款或内容模块停用、恢复和废弃仅由全局管理员和分组管理员执行；执行前必须进行影响分析、二次确认并记录审计。
- 条款或内容模块停用或废弃后，已发布且已锁定该模块版本的模板仍按发布时锁定内容生成；模块停用或废弃只阻止后续新的模板发布候选引用该模块版本。
- 如果业务需要立即停止使用包含问题条款或内容模块的已发布模板，必须通过停用对应模板或发布版本来阻断生成和 API 调用。
- 条款或内容模块停用或废弃前的影响分析必须覆盖引用模板、引用发布版本、default 路由影响、近期调用摘要、是否需要停用模板或发布版本，以及可替代模块版本建议。

### 2.9.3 模板可验证性资产 Template Verifiability Assets

模板可验证性资产用于在模板提交审批、审批和发布前证明发布候选可生成、可检查、可追溯。

已确认资产包括：

- 测试数据集。
- 测试生成记录。
- 生成预览。
- 变更差异摘要。
- 发布前检查清单。
- 审批摘要。

已确认规则：

- 测试数据集用于驱动 DOCX/PDF 测试生成、生成预览、变量 Schema 校验和规则校验。
- Template Test Data Set 由全局管理员、分组管理员、母版设计人员和模板编排人员按权限范围创建、编辑、复制、派生和删除；测试人员和审批人员仅在测试、审批材料中查看和判定，不获得维护权限。
- 同一 Template 或 Release Candidate 支持多组命名 Template Test Data Set；每组测试样例默认绑定单个 Template 或 Release Candidate，可标记场景名称、必选或可选、覆盖标签和数据集版本；同组内可复制或派生，跨模板或跨组共享测试数据集库不属于 v1 已确认范围。
- Template Test Data Set Draft 在关联 Test Generation Record、Template Test Record、Approval Material 或 Release Evidence 前可编辑；一旦进入证据链即锁定不可变，后续修改必须复制或派生新数据集版本并重新执行测试生成、测试判定和后续审批发布流程。
- Template Test Data Set 默认使用脱敏或合成数据；确需使用敏感测试值时，必须按 Sensitive Data Classification 保护，审计、摘要、Preview Artifact 和 Release Evidence 仅保存数据集版本、`variablesHash` 和非敏感统计，不保存模板变量测试值明文。
- Batch Test Summary 记录同一 Release Candidate 上多组测试样例的批量测试结果，至少包含参与样例清单、必选/可选标记、逐样例测试生成记录引用、Preview Artifact 引用、警告摘要、阻断项摘要、覆盖率摘要、批量状态和非敏感统计。
- Sample Coverage Summary 记录单组测试样例覆盖率；Template Coverage Summary 汇总当前发布候选的综合覆盖率。覆盖维度至少包括变量取值与必填字段、条件/循环/规则分支、锚点/章节、表格等受控组件和 DOCX/PDF 输出格式。
- Release Candidate 需要至少一组必选测试样例；所有必选测试样例必须测试通过且无未解决阻断项。模板综合覆盖率和关键维度覆盖率必须达到可配置阈值，低于阈值属于发布阻断项。
- 生成预览需要供模板测试和审批查看。
- 编辑态快速预览用于即时反馈，不作为测试、审批或发布的权威证据；如果展示快速预览，界面需要明确其非权威状态和与最终渲染产物的关系。
- 测试、审批和发布必须以最终 DOCX/PDF 渲染链路产生的 Preview Artifact 和最终产物引用为权威证据。
- Preview Comparison Summary 记录预览与最终 DOCX/PDF 产物的并排对比结果，至少包含参与对比的产物引用、输出格式、页/锚点/章节/组件定位、差异风险级别、关联保真警告或阻断项引用和非敏感摘要。
- 预览与最终产物差异需要按风险进入保真警告或发布阻断项；影响文档语义、合规结果、签章/盖章、附件完整性、关键表格可读性或条款编号语义的差异不得只作为提示处理。
- 变更差异摘要至少覆盖模板内容、锚点使用、模板变量、编排规则和发布候选 API 契约摘要变化。
- 发布前检查清单必须覆盖锚点完整性、变量 Schema 合法性、规则执行边界、测试生成结果、生成预览、变更差异摘要、审批摘要和阻断项状态。
- 锚点完整性检查确认模板引用的母版锚点存在；缺失锚点属于阻断项。
- 变量 Schema 校验确认模板变量定义、类型、必填性、枚举、嵌套结构和默认值等规则与发布候选契约一致；校验失败属于阻断项。
- 规则执行边界校验确认计算表达式、聚合函数、条件显示、循环渲染和跨字段校验只基于 API 入参和模板内配置，不允许调用外部数据/API；越界规则属于阻断项。
- DOCX/PDF 预览和渲染保真采用阻断项与警告项分级；严重保真问题阻断发布，较低风险保真问题作为警告进入测试、审批或发布摘要。
- 发布阻断项包括不支持节点、变量引用缺失、样式引用缺失、条款或内容模块引用缺失、表格组件引用缺失、母版锚点引用缺失、编号断裂且影响条款语义、表格无法可靠渲染、签章/盖章位置异常和 PDF 转换失败。
- 样式未通过母版审核、不在可访问母版样式目录内、不适用于节点类型、有限直接格式取值不在白名单内或无法稳定渲染，均属于发布阻断项。
- 粘贴清洗存在未解决阻断项，或不可识别内容仍以原始 Word、HTML、CSS 或二进制片段形式存在于发布契约中，均属于发布阻断项。
- 警告项用于提示不直接破坏文档语义或合规结果的较低风险保真问题；v1 不设置静默忽略阈值，任一已确认低风险保真问题都必须形成警告。
- 警告项不会仅因数量自动阻断发布；如果问题影响文档语义、合规结果、签章/盖章、二维码/条码、附件完整性、变量或引用合法性、关键表格可读性或条款编号语义，则应归为阻断项。
- v1 已确认警告项包括可选内容为空但可安全隐藏或显示空状态、低风险分页或换行差异、低风险表格跨页、受控样式回退或格式归一化、普通图片等比缩放微调。
- 警告展示必须包含稳定 `warningCode`、`messageKey`、影响位置摘要、检测结果摘要和处理建议；不得包含模板变量原值、粘贴原文、客户数据、完整请求体或生成文档敏感内容。
- 警告需要进入测试生成记录、审批材料、发布前检查清单、发布摘要、审计摘要和 API 生成成功响应。
- Fidelity Warning Display Summary 采用摘要条加筛选列表，摘要条至少记录警告总数、按 `warningCode` 分布和是否存在阻断项；列表项记录 `warningCode`、影响位置摘要、检测结果摘要、处理建议、关联 Preview Artifact 或输出产物和已查看状态。
- Fidelity Warning Display 支持按 `warningCode`、锚点/章节/组件位置、预览/DOCX/PDF 产物和已查看/未查看状态筛选；详情仅展示非敏感摘要。
- Template Test Record、Approval Summary 和 Release Summary 必须记录保真警告摘要已查看确认；该确认不要求逐条解决警告，警告数量不自动阻断发布。
- v1 复杂编号只支持平台或已审核母版样式目录确认的受控多级编号方案；条件块和循环块渲染后必须确定性重排编号，重复编号、跳号、条款编号语义变化或交叉引用失效均阻断发布。
- v1 表格组件采用结构化表格组件模型，支持列 schema、表头、跨页重复表头、循环行、简单合计/页脚和受控列宽；嵌套表、浮动表、绝对定位表、无法稳定跨页的合并单元格或无法保持关键金额/条款可读性的表格均阻断发布。
- v1 签章/盖章仅确认受控占位和印章/签名图片资产，不包含密码学电子签名、电子印章或验签能力；签章/盖章必须位于授权区域内，不得重叠、裁切、越界或在 DOCX/PDF 中不可见，否则阻断发布。
- PDF 转换验收采用语义保真优先基线；不得出现缺页、异常空白页、内容裁切、关键内容丢失、签章/盖章不可见、二维码/条码不可读、关键表格不可读或附件引用丢失。低风险视觉差异按已确认警告项处理。
- 编号、表格、签章/盖章和 PDF 转换验收结果必须进入测试生成记录、审批材料、发布前检查清单、发布摘要和审计摘要；发布前检查存在阻断项时不得发布。
- 已发布模板运行期生成对外只返回生成成功警告或生成失败错误；API 契约和响应不得暴露内部渲染诊断明文、模板变量原值、客户数据或完整生成内容。
- v1 结构化富文本和片段编辑能力仅用于文档生成主线，包括模板正文、结构化文档片段、条款或内容模块，以及已确认的受控表格、图片、二维码/条码、签章/盖章和附件清单节点。
- v1 不将该编辑能力扩展到通知模板、审批意见模板、测试意见模板或其他非文档生成业务文本场景；未来如需复用，必须单独确认对象模型、权限、生命周期、审批、审计和输出边界。
- Template Anchor Binding 必须引用已审核 Master Anchor Catalog 中的稳定 `anchorId`，并在 Template Release Version 中锁定 Master Document Version 和 Anchor Catalog Version；不得使用 Word 名称、位置或模糊匹配作为发布契约。
- 每个 `anchorId` 在同一 Template Release Candidate 中最多一个绑定；绑定对象承载一个有序 Structured Content Tree，内容类型必须符合锚点允许范围，绑定校验状态至少包括 `VALID`、`MISSING_ANCHOR`、`DUPLICATE_BINDING` 和 `INCOMPATIBLE_CONTENT_TYPE`；后三者均属于发布阻断项。
- Render Profile 是发布锁定的受控渲染配置，覆盖样式、编号、表格分页、图片缩放、PDF 转换和保真策略；API Request 不得以任意渲染参数覆盖 Template Release Version 的 Render Profile。
- Preview Artifact 是最终 DOCX/PDF 渲染链路产生的可追溯预览记录，至少包含 `previewId`、`testDataSetId`、`variablesHash`、`outputFormat`、`renderProfileVersion`、`status`、`documentId` 或预览引用、最终产物引用、警告摘要、阻断项摘要、对比摘要、创建时间、更新时间和非敏感摘要；Preview Artifact 状态集合为 `ACCEPTED`、`PROCESSING`、`SUCCEEDED`、`FAILED` 和 `EXPIRED`。
- Template Test Record、Approval Summary 和 Release Summary 按输出策略锁定最终 DOCX/PDF 产物引用、Preview Artifact 引用、Preview Comparison Summary 和非敏感摘要；预览文件、最终产物和并排对比视图按权限受控查看，审计摘要不得保存模板变量原值、客户数据、完整请求体或完整生成内容。
- 发布前检查发现阻断项时，模板不得发布。
- 模板可验证性资产需要纳入审计或可追溯记录；不得记录模板变量测试值中的敏感明文，敏感内容按审计脱敏规则处理。

### 2.9.4 模板协作待办 Template Collaboration Work Item

Template Collaboration Work Item 用于站内待办和状态提示，不是 Template Lifecycle Status。

已确认规则：

- v1 协作通知采用站内待办和状态提示，不外发邮件或即时通讯消息，不新增通知模板编辑能力。
- 协作待办触发类型包括提交测试、测试不通过或回到草稿、提交审批、审批不通过或回到草稿、审批通过进入待发布、自动超时升级。
- 提交测试生成模板所属组测试人员角色队列待办；提交审批生成模板所属组审批人员角色队列待办；测试或审批退回草稿生成提交人或模板编排人员整改待办；待发布生成具备发布权限角色的发布待办；超时升级生成对应分组管理员待办。
- 协作待办按模板所属组和角色队列分配，不默认指定个人；待办展示不得授予额外模板编辑、测试判定、审批判定或发布权限。
- 测试待办、审批待办、待发布待办和整改待办支持自动超时升级；超时阈值配置由全局默认和组级覆盖组成，全局管理员维护全局默认，分组管理员在被授权组范围内维护组级覆盖。
- 自动超时升级只生成分组管理员的站内升级待办和状态提示，不自动通过、不自动拒绝、不自动退回、不改变模板状态，也不产生代理审批。
- 分组管理员收到超时升级后如需例外干预测试或审批判定，仍必须遵循原因、二次确认和单独审计标记规则。
- 协作待办、状态提示和超时升级仅展示非敏感摘要，不得包含模板变量值、客户数据、完整生成内容或敏感明文。
- 协作待办创建、解决、超时升级和超时阈值配置变更必须记录审计摘要。

### 2.10 开发版本 Development Version

开发版本用于记录模板变更历史。

已确认规则：

- 开发版本用于记录 change history。
- 开发版本不直接锁定 API 服务。

### 2.11 发布版本 Release Version

发布版本用于锁定 API 服务。

已确认规则：

- 发布版本用于锁定 API 服务。
- 已发布版本不允许修改。
- 同一模板不同发布版本需要共存。
- API 路由需要支持显式发布版本路径；调用方通过路径选择目标模板和发布版本。
- API 需要支持 default 路径；default 路径由 API 管理配置显式路由到某个未停用发布版本。
- default 路径不得隐式指向最新版本，必须由全局管理员或分组管理员在 API 管理中显式配置目标发布版本。
- default 路径目标版本变更需要审计，并应提供影响预览。
- API 契约展示 default 路径时，需要展示 default 路径、当前目标发布版本、目标版本状态、最近更新时间、最近操作人和对应显式版本路径。
- default 路径目标版本变更只支持立即生效，不支持未来定时生效或待生效变更。
- default 路径目标版本变更不主动通知调用方或管理员，仅记录审计；调用方通过 API 契约查看当前 default 目标版本。
- default 路径目标版本变更影响预览至少包含当前目标版本与候选目标版本、授权调用方范围、近期调用量摘要、契约差异摘要和幂等影响提示。
- default 路径目标版本回滚按一次新的受控变更处理，需要重新选择历史目标发布版本，执行影响预览并记录审计；回滚同样立即生效且不主动通知。
- 版本级别支持停用。
- 单个发布版本停用只影响该发布版本，其他未停用发布版本仍可按模板级 API 管理配置调用。
- API 授权只到模板级别；调用方获得模板权限后，可以调用该模板下所有未停用的发布版本。
- 发布版本号采用语义化版本，例如 1.0.0、1.1.0、2.0.0。
- 发布时由模板编排人员或管理员选择版本变更级别。

### 2.12 动态 API Dynamic API

动态 API 是模板发布后生成的文档生成调用能力。

模板发布后需要生成：

- 接口地址。
- 请求参数 Schema。
- 响应 Schema。
- 字段校验规则。
- 示例请求/响应。
- 错误码说明。
- 可调用版本列表。

已确认规则：

- API 需要根据模板动态生成。
- API 调用权限是模板级别。
- API 分环境。
- API 通过环境变量读取当前环境。
- API 路径统一采用 `/api/{environment}/v1` 前缀；平台运行时仍通过环境变量读取当前部署环境，并校验路径中的 `{environment}` 与当前部署环境一致。
- API 需要支持批量生成。
- API 路由需要支持显式发布版本路径；调用方通过路径选择目标模板和发布版本。
- API 需要支持 default 路径；default 路径由 API 管理配置显式路由到某个未停用发布版本。
- 显式发布版本单笔生成路径为 `/api/{environment}/v1/templates/{templateId}/versions/{releaseVersion}/generate`。
- default 单笔生成路径为 `/api/{environment}/v1/templates/{templateId}/default/generate`。
- 显式发布版本批量生成路径为 `/api/{environment}/v1/templates/{templateId}/versions/{releaseVersion}/batch-generate`。
- default 批量生成路径为 `/api/{environment}/v1/templates/{templateId}/default/batch-generate`。
- 异步任务查询路径为 `/api/{environment}/v1/templates/{templateId}/tasks/{taskId}`。
- 下载地址取文件路径为 `/api/{environment}/v1/documents/{documentId}/download`；下载时仍需要通过 `documentId` 关联模板并执行模板级二次授权。
- API 契约查看路径为 `GET /api/{environment}/v1/templates/{templateId}/contract`。
- 可调用版本列表路径为 `GET /api/{environment}/v1/templates/{templateId}/versions`；该接口返回当前授权视角下可调用的发布版本列表，不作为后台版本管理列表。
- API v1 请求头字段确认为 `X-Api-Credential-Id`、`X-Api-Credential-Secret`、`X-Access-Account`；可选追踪请求头为 `X-Trace-Id`。
- `X-Trace-Id` 传入时平台沿用该值作为响应和审计中的 `traceId`；未传入时由平台生成 `traceId`。
- 批量请求支持批次级统一输出和加密配置，也允许单笔记录单独覆盖输出格式、输出模式和加密参数。
- 批量请求中每笔记录的单独覆盖都必须受模板级 API 管理配置约束，不能绕过输出方式、批量上限或动态加密能力限制。
- API 管理配置当前按模板级绑定；一个模板对应一组 API 管理配置，适用于该模板下所有未停用的发布版本；v1 不提供发布版本级 API 管理配置覆盖机制。
- 发布版本继续锁定模板内容、变量、规则和发布版本契约；API 管理配置作为调用侧策略独立维护，不要求重新发布模板。
- 可调用版本列表从模板下发布版本生成；模板停用或废弃时所有发布版本不可调用，单个发布版本停用时仅该版本不可调用；模板或发布版本恢复后，恢复对象重新进入可调用候选范围，但仍受模板状态、发布版本状态和模板级 API 管理配置约束。
- DOCX 和 PDF 输出均需要支持根据 API 参数动态加密；加密能力必须受 API 管理配置控制。
- API 不需要按输出格式单独控制权限；调用方获得模板权限后，可以按 API 管理配置生成 DOCX 或 PDF。
- API 不需要按输出模式单独控制权限；调用方获得模板权限后，可以使用 API 管理配置允许的同步文件流、同步下载地址、异步任务或批量生成模式。
- 批量 API 失败策略按调用模式区分：异步批量允许部分成功并返回成功/失败明细；同步批量要求全部成功或全部失败。
- 批量 API 采用全局默认上限 + API 管理可覆盖：默认同步批量最多 100 条，默认异步批量最多 10,000 条；具体 API 管理配置可以配置更低上限。
- 批量请求中每笔记录必须传入 `items[].itemId`，且同一批次内必须唯一。
- 同一批次出现重复 `items[].itemId` 时，整批请求校验失败，返回 `400 ITEM_ID_DUPLICATED`，不创建批次或异步任务。
- 所有文档生成类 API 必须传入 `idempotencyKey`；异步任务查询和下载地址取文件不适用该要求。
- 所有文档生成类 API 必须传入 `requestId`，用于调用方业务追踪。
- `requestId` 用于调用方业务追踪，`idempotencyKey` 用于重复提交识别。
- 幂等唯一性范围为调用方、环境、模板和解析后的发布版本。
- 幂等记录保留 7 天；记录过期后，同一 `idempotencyKey` 可按新请求处理。
- default 路径目标版本切换后，如果重复提交命中旧幂等记录，应返回幂等冲突错误，不按新的 default 目标版本生成文档。
- 相同 `idempotencyKey` 命中已成功的同步文件流请求时，允许重放原文件流结果与对应响应头元数据。
- 相同 `idempotencyKey` 命中已成功的同步下载地址请求时，优先返回原下载地址；若原下载地址已过期且结果仍在保留期内，重复命中响应可重新签发新的下载地址。
- 相同 `idempotencyKey` 命中单笔异步请求时，重复命中响应返回原任务完整状态对象，而不是仅返回 `taskId`。
- 相同 `idempotencyKey` 的原请求失败后，仅系统类临时故障且 `retryable=true` 的场景允许自动重执行；业务校验、授权和策略类失败不自动重执行，重复命中返回原失败结果。
- 幂等响应字段在 JSON 响应中统一放入 `metadata`；同步文件流响应通过响应头表达核心幂等元数据。
- 生成类 API 的成功和错误响应都回显 `idempotencyKey`；同步文件流通过响应头回显。
- 重复命中成功场景固定返回 `originalRequestAt`，采用 ISO 8601 带时区偏移格式。
- 幂等处理状态枚举采用 `IDEMPOTENCY_NEW`、`IDEMPOTENCY_REPLAYED`、`IDEMPOTENCY_CONFLICTED`。
- 幂等冲突响应允许返回安全差异摘要，但不得返回旧请求或新请求的业务变量原值、加密密码、完整请求体或敏感配置明文。
- 幂等冲突安全差异摘要对象名采用 `error.idempotencyConflict`，可包含 `conflictType`、`conflictFields`、`originalRequestAt`、`originalResolvedReleaseVersion`、`requestHash`、`variablesHash`、`itemsHash`。
- `error.idempotencyConflict.conflictType` 基线值包括 `REQUEST_SEMANTICS_MISMATCH` 和 `DEFAULT_ROUTE_CHANGED`。
- 幂等记录过期后复用同一 `idempotencyKey` 按新请求处理；API 响应不提示历史复用信息，仅在审计中标记。
- 过期 `idempotencyKey` 复用审计字段包括 `reusedExpiredIdempotencyKey`、`previousIdempotencyExpiredAt`、`previousRequestAt`、`previousResolvedReleaseVersion`。
- 批量请求使用批次级 `idempotencyKey` 识别整批重复提交；`items[].itemId` 只用于单笔明细、审计和失败项定位，不替代批次级 `idempotencyKey`。
- 同步批量中任一记录因参数校验或 API 管理策略失败时，整批失败且不生成任何文件；响应需要返回每笔失败明细，并按非重试幂等结果记录，重复提交同一 `idempotencyKey` 时重放该失败结果。
- 异步批量部分成功后的失败项重试必须使用新批次和新的 `idempotencyKey`；新批次只提交需要重试的失败项，并通过 `originalBatchId` 或等效关联字段关联原批次，原批次结果不被扩展或改写。
- 同步文件流响应中，文件内容放在响应体，核心元数据通过响应头承载；核心元数据包括 `auditId`、`traceId`、`requestId`、`idempotencyKey`、`idempotencyStatus`、`documentId`、`templateId`、`routeType`、`resolvedReleaseVersion`、`output.format`、`output.mode`。
- 同步下载地址和异步结果下载地址固定有效期为 15 分钟，不允许通过 API 管理配置调整有效期。
- 下载时需要二次授权，校验 API 凭证、AD Group、模板级授权、下载地址有效期和结果有效性。
- 下载时不重新校验发布版本是否仍可调用；已生成结果不会因为发布版本后续停用或 default 路径目标变更而在下载阶段失效。
- 下载地址在 15 分钟有效期内允许多次下载；不允许 API 管理配置覆盖为一次性下载。
- 下载响应需要返回 `download.expiresAt` 和 `download.oneTime`；当前 v1 策略下 `download.oneTime` 固定为 `false`。
- `download.expiresAt`、`task.acceptedAt`、`task.updatedAt`、`task.expiresAt`、`originalRequestAt` 等 API 时间字段采用 ISO 8601 带时区偏移格式，例如 `2026-06-03T16:30:00+08:00`；不得返回无时区本地时间或纯 Unix 时间戳。
- `download.url` 在 API 响应中返回对当前授权调用方可用的地址；日志、审计、管理界面和契约示例展示下载地址时必须脱敏，不展示完整可用地址。
- 下载地址取文件接口在地址过期后不重新签发下载地址；调用方需要通过新的生成请求或命中可复用幂等记录获取可用下载地址。
- 异步任务查询需要返回任务状态、响应元数据、成功结果或统一错误明细；异步批量任务需要返回批次汇总和单笔成功/失败明细。
- 异步任务查询接口可选接受 `requestId` 作为附加追踪标识；该字段不参与任务定位和幂等判断，仅用于审计关联与排查。
- 异步任务受理响应需要返回 `task.queryPath`，值为任务查询相对路径，不是免认证或签名地址；后续查询仍需 API 凭证、AD Group 和模板级授权。
- 异步任务 v1 支持受控取消，取消路径采用 `POST /api/{environment}/v1/templates/{templateId}/tasks/{taskId}/cancel`；仅未完成且未过期的任务可取消，已完成、已失败、已过期或其他不可取消状态返回 `409 ASYNC_TASK_CANCELLATION_NOT_ALLOWED`。
- 异步任务取消成功后的最终状态为 `CANCELLED`；取消后的任务不返回已生成结果、下载地址或异步批量单笔成功结果，即使取消前已有部分单笔生成完成。
- 异步任务查询不返回进度百分比；单笔任务通过状态和时间字段表达进展，异步批量任务通过 `batch.summary` 返回总数、已处理数、成功数、失败数和跳过数等进度摘要。
- 异步任务状态集合确认为 `ACCEPTED`、`PROCESSING`、`SUCCEEDED`、`FAILED`、`PARTIAL_SUCCEEDED`、`EXPIRED`、`CANCELLED`；`PARTIAL_SUCCEEDED` 仅用于异步批量任务。
- 异步任务和生成结果默认保留 7 天。
- 生成结果 7 天到期清理前不主动通知调用方或管理员，仅记录清理审计。
- API 错误模型采用细分 `error.code` + `error.category` 分组；`error.code` 使用稳定细分错误码，`error.category` 用于调用方按大类处理。
- 所有 API 错误响应必须返回 `error.retryable`，明确重复提交或重试是否允许。
- API 错误响应的 `error.message` 使用英文业务可读消息，并返回 `error.messageKey` 供调用方进行多语言映射。
- API 错误类别采用 10 类固定集合：`AUTHENTICATION`、`AUTHORIZATION`、`VERSION_ROUTING`、`API_POLICY`、`IDEMPOTENCY`、`VALIDATION`、`TEMPLATE_CONTRACT`、`GENERATION`、`ENCRYPTION`、`BATCH`。
- API v1 基线错误码清单需要覆盖认证、授权、版本路由、API 管理策略、幂等、参数校验、模板契约、生成、加密、批量、异步任务和下载取文件场景；完整清单、默认 `retryable` 和英文消息以 [API 文档索引](../api/README.md) 与 [OpenAPI v1](../api/openapi-v1.yaml) 为准。
- `error.messageKey` 命名规则采用 `api.error.<category>.<camelCaseCode>`，例如 `api.error.versionRouting.defaultRouteNotConfigured`。
- `error.message` 采用简洁、业务可读且不泄露敏感信息的英文消息，不暴露 API 凭证、密码、内部配置或未授权资源细节。
- `error.message` 保持错误码级别的通用安全英文文案；同一 `error.code` 不因具体业务场景临时返回不同 `message`。
- 不为错误响应新增 `resolutionHint`、`developerMessage` 等提示字段；更具体的业务说明通过字段级错误 `fieldErrors[].message`、安全差异摘要或 API 契约示例表达。
- 错误响应示例采用重点场景覆盖，不要求每个错误码都提供独立示例；重点覆盖授权与 AD Group、版本与 default 路由、API 管理策略、异步与下载结果、生成与加密失败、批量单笔失败。
- 字段级错误路径使用点路径和数组下标，例如 `variables.customerName`、`items[0].variables.amount`。
- 字段级错误原因 `fieldErrors[].reason` 采用通用枚举集合：`REQUIRED`、`INVALID_TYPE`、`INVALID_FORMAT`、`OUT_OF_RANGE`、`TOO_LONG`、`TOO_SHORT`、`ENUM_NOT_ALLOWED`、`PATTERN_MISMATCH`、`RULE_FAILED`、`DUPLICATED`、`UNKNOWN_FIELD`。
- 批量部分失败以 `result.batch.items[].error` 承载单笔错误；顶层 `error` 仅用于整批失败或整个请求失败；整批失败需要返回每笔明细时，单笔明细嵌入 `error.items`。
- API 错误响应 HTTP 状态码用于表达错误大类，调用方判断精确失败原因仍以稳定 `error.code` 为主。
- 认证错误返回 401；授权错误返回 403；AD Group 解析失败返回 503。
- 请求结构、缺失字段、格式类校验错误返回 400；模板变量或业务规则校验失败返回 422。
- 输出格式、输出模式、批量上限、加密能力等 API 管理策略拒绝返回 400。
- 资源不存在返回 404；下载地址、异步任务或生成结果过期返回 410；版本、模板、default 目标不可用或异步任务不可取消返回 409。
- 幂等冲突和不可重试重复提交返回 409；幂等存储临时不可用返回 503。
- 模板契约异常、锚点缺失、文档生成失败、PDF 转换失败、加密处理失败或整批处理失败返回 500；生成服务临时不可用返回 503；生成超时返回 504。
- 异步任务受理成功返回 202。
- 异步批量部分成功返回 200，并通过 `result.batch.items[].status` 与 `result.batch.items[].error` 表达单笔失败，不使用顶层 `error` 表达部分失败。
- API 调用需要审计记录。
- API 契约信息和可调用版本列表可由管理员、模板编排人员、以及被授权的 API 调用方查看。
- v1 不建设独立开发者门户；后台 API 契约页调用方视图作为调用方契约入口。
- 后台 API 契约页调用方视图需要展示授权模板的契约版本对比、错误码说明、调用示例、可调用版本列表、API 策略摘要和调用方自身 API 凭证非敏感状态。
- 契约版本对比由后台 API 契约页调用方视图基于已授权可见的现有契约信息、可调用版本、请求 Schema、响应 Schema、错误码、API 策略、路由/default 目标和示例计算展示；v1 不在 `ContractResponse` 中新增专门的契约版本对比字段。
- 契约版本对比只展示非敏感摘要，不展示模板变量原值、客户数据、完整请求体、API 凭证 secret、加密密码、完整生成内容或完整审计明细。
- API 调用方 v1 不能自助创建、轮换或吊销 API 凭证；只能查看自身凭证状态、到期时间和指纹摘要等非敏感信息，API 凭证创建、轮换和吊销仍由全局管理员或分组管理员按权限执行。
- API 调用方只能查看自己被授权模板的 API 契约。

### 2.13 API 凭证 API Credential

API 凭证用于 API 调用认证，是双重认证的一部分。

已确认规则：

- API 采用 API 凭证 + AD Group 双重认证授权。
- API 凭证代表调用系统/应用。
- API 凭证对象是调用方级身份，可授权到多个模板 API。
- API 凭证由全局管理员和分组管理员管理。
- 全局管理员可管理全部 API 凭证，分组管理员只能管理被授权组范围内的 API 凭证。
- API 凭证创建和轮换时，secret 明文只展示一次；平台只保存不可逆摘要或指纹，不允许管理员后续重新查看 secret 明文。
- API 凭证必须设置有效期；默认有效期为 180 天，最长 365 天，管理员可设置更短有效期。
- API 凭证状态集合确认为 `ACTIVE`、`EXPIRING_SOON`、`EXPIRED`、`REVOKED`。
- `EXPIRING_SOON` 用于到期前提醒窗口，轮换状态由当前 secret 与旧 secret 宽限期表达，不新增凭证级 `ROTATING` 状态。
- API 凭证轮换时，新 secret 立即可用，旧 secret 保留 7 天宽限期；宽限期结束后旧 secret 失效。
- API 凭证吊销立即生效，阻断该凭证的所有后续 API 操作，包括新生成、异步任务查询、异步任务取消和下载取文件。
- 已受理的后台生成任务可继续完成，但调用方不能再使用被吊销凭证获取结果。
- API 凭证过期后，使用该凭证的 API 请求返回 `401 API_CREDENTIAL_EXPIRED`；API 凭证吊销后，使用该凭证的 API 请求返回 `401 API_CREDENTIAL_REVOKED`。
- API 凭证到期前 30 天、7 天和 1 天提醒全局管理员和对应分组管理员；不主动提醒 API 调用方，调用方可通过 API 契约或管理界面查看自己凭证的非敏感状态和到期摘要。
- API 凭证生命周期审计需要覆盖创建、轮换、吊销、过期、到期提醒和凭证摘要查看；审计至少记录操作者、时间、操作原因、管理范围、状态变化、到期时间、凭证标识或指纹摘要和受影响授权范围，不记录 secret 明文。
- 请求中同时识别实际访问账号。
- 实际访问账号用于读取 AD Group。

### 2.14 API 管理配置 API Management Configuration

API 管理配置用于维护模板生成 API 的调用侧配置，和模板编排严格区分。

已确认属性：

- 关联模板。
- 不绑定单个发布版本。
- API 输出方式配置。
- 批量上限配置。
- AD Group 授权集合。
- DOCX/PDF 动态加密能力配置。
- default 路径目标发布版本配置。

已确认规则：

- API 管理由全局管理员和分组管理员承担。
- 全局管理员可管理全部 API 管理配置。
- 分组管理员只能管理被授权组范围内的 API 管理配置。
- API 管理配置当前按模板级绑定；一个模板对应一组 API 管理配置，适用于该模板下所有未停用的发布版本。
- API 管理配置与模板绑定，不与单个发布版本绑定；配置变更对该模板下所有未停用发布版本生效；v1 不提供发布版本级 API 管理配置覆盖机制。
- API 管理配置变更需要审计，但不改变已发布版本锁定的模板内容、变量或规则。
- API 管理配置界面采用模板级 API 管理页，使用配置域导航 + 详情区；配置域导航固定包含 AD Group 授权、输出方式、批量上限、DOCX/PDF 动态加密能力和 default 路径目标发布版本，详情区展示当前配置摘要、候选配置编辑区、字段提示、影响预览、硬阻断和警告、保存确认动作、当前 `policyVersion`、最近更新时间、最近操作人和审计入口。
- API 管理配置字段控件基线：AD Group 授权使用可搜索 AD Group 选择器和授权范围摘要，不展示完整成员或未授权组详情；输出方式使用输出格式和输出模式勾选；批量上限使用同步/异步数值输入并展示上限含义；DOCX/PDF 动态加密使用启用开关和能力项选择，不保存加密密码；default 路径目标发布版本使用发布版本选择器，并展示版本状态、契约摘要和影响提示。
- API 管理配置按配置域独立保存；每个配置域操作动线为编辑候选配置、执行影响预览、处理硬阻断或确认警告、管理员确认立即生效；候选配置变更后必须重新执行影响预览，保存成功生成新的 `policyVersion` 和审计记录。
- API 管理配置引入 `policyVersion`；每次配置域变更成功生效后生成新的配置版本，用于契约展示、审计、影响预览和回滚关联。
- AD Group 授权、输出方式、批量上限、DOCX/PDF 动态加密能力变更均需要影响预览；default 路径目标发布版本继续遵循已确认的专门治理规则。
- API 管理配置影响预览需要区分硬阻断和警告；违反已确认策略或会导致候选配置不可生效的硬阻断必须阻止保存，风险提示类警告允许管理员确认后继续。硬阻断和警告文案采用固定结构：原因、影响、处理建议；影响信息至少包含受影响发布版本或调用方范围摘要和预期错误码。硬阻断文案必须明确无法保存，警告文案必须明确确认继续后会立即生效并记录审计。
- 其他 API 管理配置变更只支持立即生效，不支持未来定时生效或待生效变更；变更和回滚均不主动通知调用方或管理员，仅记录审计。
- API 契约按角色展示 API 管理配置：被授权 API 调用方只能查看当前可用策略摘要；管理员和模板编排人员可查看管理配置详情；API 凭证、完整 AD Group 成员、敏感值和历史密文不得暴露。
- API 管理配置展示字段 v1 基线为 `apiPolicy.policyVersion`、`apiPolicy.updatedAt`、`apiPolicy.updatedBy`、`apiPolicy.allowedOutputFormats`、`apiPolicy.allowedOutputModes`、`apiPolicy.batchLimits.syncMaxItems`、`apiPolicy.batchLimits.asyncMaxItems`、`apiPolicy.encryptionCapabilities`、`apiPolicy.adGroupAuthorizationSummary`、`apiPolicy.credentialSummary`；不得展示 API 凭证 secret、完整 AD Group 成员、未授权组详情、加密密码、历史密文或其他敏感配置明文。
- 其他 API 管理配置影响预览至少包含当前配置与候选配置差异、受影响模板及未停用发布版本、授权调用方或 AD Group 范围摘要、近期调用量摘要、可能被拒绝的输出模式/批量/加密调用摘要和预期错误码提示。
- 其他 API 管理配置回滚按一次新的受控变更处理，需要选择历史配置作为候选配置，执行影响预览，确认后立即生效，并记录审计。
- API 凭证生命周期、轮换、吊销、过期、到期提醒和相关审计策略已确认；API 凭证到期提醒是生命周期专门通知，不改变其他 API 管理配置变更不主动通知调用方或管理员的基线。
- default 路径目标发布版本属于 API 管理配置，必须显式指向一个未停用发布版本，不得隐式指向最新版本。
- default 路径目标版本变更只支持立即生效，不支持未来定时生效或待生效变更；回滚按一次新的受控变更处理。
- AD Group 授权配置属于 API 管理功能，不属于模板编排或模板提交功能。
- DOCX/PDF 动态加密配置属于 API 管理功能，不属于模板编排或模板提交功能。
- API 调用方获得模板权限后，可以调用该模板下所有未停用的发布版本，但仍受 API 管理配置约束。

- 批量上限配置变更。
- DOCX/PDF 动态加密配置变更。
- default 路径目标发布版本配置变更。

API 调用审计至少记录：

- 环境。

- 请求参数摘要。

已确认审计查看权限：

- 审计管理员可查看全部审计记录。
- 全局管理员按已确认最大权限范围查看审计记录。
- 分组管理员只能查看被授权组范围内的审计明细。
- 分组管理员不能查看未授权组审计明细，也不能查看全局审计明细。
- 母版设计人员、模板编排人员、测试人员、审批人员、API 调用方不因该角色本身获得审计查看权限。

已确认审计导出权限：

- 审计记录允许导出。
- 全局管理员和审计管理员可导出全部审计记录。
- 分组管理员只能导出被授权组范围内的审计记录。
- 母版设计人员、模板编排人员、测试人员、审批人员、API 调用方不因该角色本身获得审计导出权限。

已确认审计保留规则：

- 审计记录保留期限可配置，默认保留 5 年。
- 审计保留期限可按环境、监管或业务要求配置。

### 2.16 授权判定 Authorization Decision

授权判定用于在受保护操作执行前确认主体是否可以访问目标对象或返回目标响应。

适用入口包括：

- 文档生成 API。
- 批量生成。
- 异步任务查询。
- 异步任务取消。
- 下载取文件。
- API 契约查看。
- 可调用版本列表。
- API 管理。
- 审计查看和导出。
- 后台模板操作。
- 后台母版操作。

已确认规则：

- 授权判定在执行受保护操作或返回敏感响应前完成。
- 授权依赖不可用且没有已确认可用缓存时按 fail-closed 处理。
- 授权判定按入口类型组合校验身份、角色、分组范围、API 凭证、访问账号、AD Group、模板级授权、对象归属、环境、资源状态和 API 管理配置。
- 任务查询、任务取消和下载取文件必须通过 `taskId` 或 `documentId` 解析到关联模板，并执行模板级二次授权。
- API 契约查看和可调用版本列表必须按当前授权视角返回结果。
- API 管理、审计查看/导出、后台模板和母版操作必须同时满足角色权限、分组范围和对象归属规则。
- 授权拒绝或授权依赖失败只返回已确认的安全错误码和通用安全消息，不泄露未授权资源是否存在、未授权组详情、完整成员列表、API secret、加密密码或内部配置细节。
- 授权判定和授权拒绝需要记录安全审计摘要，包含主体摘要、入口、环境、对象范围摘要、判定结果、拒绝原因码或依赖失败原因；不得记录敏感明文。

### 2.17 敏感数据分类 Sensitive Data Classification

敏感数据分类用于约束响应、日志、审计、管理界面、API 契约展示、契约示例、导出文件和支持排查材料中的数据展示方式。

已确认分级：

- 禁止明文持久化/展示。
- 允许摘要或指纹。
- 授权响应例外。

禁止明文持久化或展示的内容包括 API 凭证 secret、DOCX/PDF 加密密码、模板变量原值、模板测试数据敏感值、完整请求体、完整下载地址、完整 AD Group 成员、未授权组详情、历史密文、敏感配置明文、内部渲染诊断明文和未授权生成文档内容；保真警告不得包含模板变量原值、粘贴原文、客户数据、完整请求体或生成文档敏感内容。

允许以摘要或指纹表达的内容包括 API 凭证标识或指纹摘要、`idempotencyKey` 摘要、请求语义 hash、`variablesHash`、`itemsHash`、加密策略摘要、AD Group 授权摘要、下载地址脱敏值、`contextSummary`、`fidelityWarnings` 非敏感摘要、`policyVersion`、`changedAreas` 和配置差异摘要。

授权响应例外仅限已确认安全场景：API 凭证创建或轮换时 secret 明文只展示一次；授权 API 响应可返回可用 `download.url`；同步文件流和下载取文件可在授权通过后返回生成文档内容；`task.queryPath` 只是相对查询路径，不授予额外访问能力。

未知或未分类字段默认按敏感处理，只能在明确确认安全后降级为摘要或可展示字段。

## 3.1 输出加密规则

已确认规则：

- DOCX 和 PDF 输出均需要支持根据 API 参数动态加密。
- 加密能力必须受 API 管理配置控制。
- DOCX/PDF 动态加密时，API 可以直接传入密码。
- API 传入的 DOCX/PDF 加密密码不落库、不进日志，只在本次生成过程中使用。
- 加密审计只记录是否启用加密和加密策略摘要。
- DOCX/PDF 动态加密配置属于 API 管理功能，不属于模板编排或模板提交功能。
- DOCX/PDF 动态加密 API 参数采用标准模型，支持 enabled、openPassword、ownerPassword、permissions。
- 加密策略摘要采用标准摘要，记录是否启用加密、输出格式、是否传入 openPassword、是否传入 ownerPassword、permissions 权限摘要。
- `encryption.enabled=true` 时，`openPassword` 必填；`ownerPassword` 可选。
- `permissions` 采用统一抽象权限枚举，按 DOCX/PDF 输出格式映射到对应加密能力；v1 权限枚举确认为 `ALLOW_PRINT`、`ALLOW_COPY`、`ALLOW_EDIT`、`ALLOW_ANNOTATE`、`ALLOW_FORM_FILL`；传入 `permissions` 时必须同时传入 `ownerPassword`。
- 不支持的权限组合、缺少必需密码、`enabled=false` 或未传 `enabled` 时仍传入加密子字段，均返回 `400 ENCRYPTION_PARAMETER_INVALID`。
- `openPassword` 和 `ownerPassword` 的密码强度基线为最少 12 字符、最长 128 字符；如果两者同时传入，二者必须不同。
- 加密参数合法但实际加密处理失败时，返回 `500 ENCRYPTION_FAILED`，`retryable=true`；错误响应、日志和审计不得返回密码、内部加密细节或敏感配置值。

## 3.2 动态 API 契约规则

已确认规则：

- 动态 API v1 请求字段命名基线采用 `output.format`、`output.mode`、`variables`、`encryption`、`requestId`、`idempotencyKey`、`items[].itemId` 和 `context`。
- `context` 采用安全白名单，v1 仅允许 `sourceSystem`、`channel`、`businessRequestId`、`upstreamTraceId`、`scenario`、`locale`；字段值均为字符串。`context` 不得包含客户姓名、证件号、账号、金额、密码、模板变量原值、完整请求体、API secret、完整下载地址或完整 AD Group 成员等敏感内容；未知 `context` 字段返回 `400 REQUEST_BODY_INVALID`。
- 模板标识和发布版本号只通过路径表达，生成请求体不得重复传入 `templateId` 或 `releaseVersion`。
- 正式 API 契约 Schema 采用 OpenAPI 3.1 YAML 维护；Markdown 文档负责解释、索引、决策背景和示例说明。
- v1 请求采用严格字段校验，契约 Schema 之外的未知字段返回 `400 REQUEST_BODY_INVALID`。
- JSON 响应采用统一 envelope：`metadata`、`result`、`error` 为顶层承载对象；同步文件流响应体只承载文件内容，核心元数据通过响应头表达。
- 幂等相关响应字段归入 `metadata`，不再作为独立顶层字段散落表达。
- 批量 JSON 响应必须按请求顺序返回全量单笔明细，每个输入对应一条明细，回显 `itemId`，并包含单笔状态、最终输出配置、加密策略摘要以及成功结果或错误信息。
- v1 API 枚举值统一采用英文 `UPPER_SNAKE_CASE`。
- 输出格式枚举确认为 `DOCX`、`PDF`；输出模式枚举确认为 `SYNC_STREAM`、`SYNC_DOWNLOAD_URL`、`ASYNC_TASK`；路由类型枚举确认为 `EXPLICIT_VERSION`、`DEFAULT_ROUTE`。
- 异步任务状态枚举确认为 `ACCEPTED`、`PROCESSING`、`SUCCEEDED`、`FAILED`、`PARTIAL_SUCCEEDED`、`EXPIRED`、`CANCELLED`；批量单笔状态枚举确认为 `SUCCEEDED`、`FAILED`、`SKIPPED`。
- `templateId` 采用可读稳定模板键：`TPL-` 前缀 + 英文、数字和连字符；发布后不可修改，不得包含客户、个人、账号、金额或其他敏感业务信息。
- `taskId`、`batchId`、`documentId` 采用资源前缀 + 不透明随机 token：`TASK-`、`BATCH-`、`DOC-`；token 不得承载日期、序号、模板、客户或业务变量含义。

## 4. 模板状态模型

模板状态需要包括：

- 草稿。
- 测试中。
- 测试通过。
- 待审批。
- 待发布。
- 已发布。
- 已停用。
- 已废弃。

### 4.1 产品状态 ↔ 实现映射（2026-06-24）

PRD §7 使用中文产品状态名；运行时 `TemplateLifecycleStatus` 枚举与 API
`approvalSubState` 字段实现如下映射。**不得**将 PRD 产品状态静默重命名；实现通过
子状态表达 PRD「测试通过」与「待审批」两阶段。

| PRD 产品状态 | `TemplateLifecycleStatus` | `approvalSubState` | 说明 |
| --- | --- | --- | --- |
| 草稿 | `DRAFT` | — | |
| 测试中 | `TESTING` | — | |
| 测试通过 | `APPROVAL` | `PENDING_SUBMIT` | 测试判定通过后进入；等待具备权限角色提交审批 |
| 待审批 | `APPROVAL` | `PENDING_DECISION` | `SUBMIT_FOR_APPROVAL` 之后；等待审批判定 |
| 待发布 | `PENDING_RELEASE` | — | 审批通过后 |
| 已发布 | `PUBLISHED` | — | |
| 已停用 | `STOPPED` | — | |
| 已废弃 | `DEPRECATED` | — | |

实现类：`TemplateLifecycleService`、`TemplateService.resolveApprovalSubState`、
`TemplateDetailView.approvalSubState`。完整发布门禁、受控意见表单与覆盖率阈值仍属
P19 范围（见 [P5 薄切片边界](../plan/detail/P5-lifecycle-governance.md)）。

已确认状态规则：

- v1 不新增发布前或发布后中间状态；发布前检查、生成预览、测试生成、渲染任务、阻断项和失败原因通过 Template Test Record、Preview Artifact、Pre-release Checklist、Release Summary 和 Audit Summary 表达，不升级为 Template Lifecycle Status。
- 模板必须经过测试人员测试。
- 模板必须经过审批人员审批。
- 测试人员按分组配置。
- 审批人员按分组配置。
- 提交测试只能从草稿状态或测试通过状态发起；提交测试后模板进入测试中状态。
- 正常测试判定由测试人员执行；分组管理员只在异常处理或补救场景中，作为例外管理员在被授权组范围内执行测试通过/不通过判定。
- 正常审批判定由审批人员执行；分组管理员只在异常处理或补救场景中，作为例外管理员在被授权组范围内执行审批通过/不通过判定。
- 分组管理员例外干预测试或审批判定时，必须填写原因、执行二次确认，并在审计中添加单独的管理员例外干预标记。
- 测试通过后，模板进入测试通过状态，不自动进入待审批状态。
- 测试通过状态下，需要由具备提交审批权限的角色手动提交审批；提交审批后模板进入待审批状态。
- 测试判定需要记录结构化结果和测试意见；测试意见采用受控表单模板，不复用结构化富文本编辑器。测试通过需要确认测试证据摘要、批量测试摘要、覆盖率摘要、生成预览摘要和保真警告摘要，可填写补充说明；测试不通过必须填写原因分类、影响范围和修复建议，并记录关联测试数据集和生成预览摘要。
- 测试不通过后，模板回到草稿状态。
- 审批判定需要记录结构化结果和审批意见；审批意见采用受控表单模板，不复用结构化富文本编辑器。审批通过必须填写理由摘要并确认关键证据摘要；审批不通过必须包含退回原因分类、影响范围和整改要求，并关联测试记录、变更差异摘要和发布前检查清单摘要。
- Risk Prompt Copy 配置由系统默认文案和可选组级覆盖组成；全局管理员维护全局默认，分组管理员在被授权组范围内维护覆盖文案和原因分类。风险提示至少覆盖未解决阻断项、保真警告、覆盖率低于阈值、预览对比差异、API 契约或影响范围变化和管理员例外干预；配置变更必须记录审计。
- 审批不通过后，模板回到草稿状态。
- 测试不通过或审批不通过回到草稿后，原测试记录和审批记录作为历史保留；再次提交发布候选时必须完整重新提交测试和审批，生成新一轮测试记录和审批记录。
- 审批通过后，模板进入待发布状态，不直接发布。
- 待发布状态下，由具备发布模板权限的角色手动发布；当前已确认具备发布权限的角色以权限矩阵为准。
- 待发布状态下执行发布前，必须展示发布摘要并进行二次确认；发布摘要至少包含发布版本号、变更差异摘要、批量测试摘要、覆盖率摘要、最终产物引用、预览对比摘要、发布前检查清单结果、API 契约摘要和影响范围。
- 待发布状态下发布前检查发现阻断项时，不新增发布失败状态；模板保持待发布但不得发布，发布前检查清单记录阻断项。若修复需要修改模板内容、依赖版本、测试数据集或渲染配置，必须回到草稿或创建新的开发版本，并重新执行测试、审批和发布流程。
- 模板审批为一级审批。
- v1 引入站内协作待办超时升级，但超时升级不改变模板状态、不自动判定、不产生代理审批；任何代理审批、自动判定或自动状态变更机制都需要单独确认。
- 提交审批和审批判定需要能查看测试记录、批量测试摘要、覆盖率摘要、生成预览、最终产物引用、预览对比摘要、变更差异摘要和发布前检查清单结果。
- 发布模板前必须执行发布前检查清单；如存在阻断项，模板不得发布。

## 5. 停用与废弃

已确认规则：

- 模板停用后，所有版本都停用。
- 版本级别也支持停用。
- 单个发布版本停用只影响该发布版本，其他未停用发布版本仍可按模板级 API 管理配置调用。
- 发布版本停用由全局管理员和分组管理员执行；全局管理员可停用全部发布版本，分组管理员可停用被授权组范围内模板的发布版本。
- 已停用表示暂时不可用，可恢复。
- 已停用后 API 不可调用，管理员可恢复。
- 模板和单个发布版本均允许从已停用状态恢复；恢复需要具备权限的管理员执行影响预览、二次确认并记录审计。
- 恢复影响预览至少包含受影响 API 调用方或 AD Group 范围、default 路由影响、恢复后可调用版本变化和近期调用摘要。
- 模板恢复后，模板下未停用且未废弃的发布版本按模板级 API 管理配置重新进入可调用候选范围；单个发布版本恢复后，仅该发布版本重新进入可调用候选范围，仍受模板状态和 API 管理配置约束。
- 已废弃表示永久下线，不可恢复。
- 已废弃后 API 永久不可调用，只保留历史和审计。
- 模板废弃前必须先停用且无可调用发布版本；废弃需要二次确认、填写原因、影响预览并记录审计，不强制主动通知调用方。
- 模板废弃由全局管理员和分组管理员执行；全局管理员可废弃全部模板，分组管理员可废弃被授权组范围内模板。

## 6. 环境迁移规则

已确认规则：

- 模板可以从测试环境手动导出。
- 模板导出/导入由管理员和模板编排人员执行；全局管理员可导出/导入全部模板，分组管理员可导出/导入被授权组范围内模板，模板编排人员可导出/导入自己负责的模板。
- 导入到生产环境后，需要从草稿阶段重新走一遍流程。
- 导入生产环境时，不能直接导出为最终发布版本。
- 导入生产遇到已有相同模板 ID 时，保留模板 ID，并在目标环境创建新的开发版本；模板发布版本号在后续发布时再由模板编排人员或管理员选择。
- 导入生产不重新生成模板 ID 或 API 地址；导入后的模板仍需从草稿重新经过测试、审批、待发布和发布流程后才可形成新的发布版本。
